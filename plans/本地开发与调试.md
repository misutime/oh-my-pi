# oh-my-pi 本地开发与调试指南

## 环境要求

- **Bun ≥ 1.3.14**

> bun -v
> bun upgrate

- macOS / Linux（Windows 部分支持）
- **Rust `nightly-2026-04-29`**（仅修改 `crates/pi-natives/` 时需要；`rustup` 会自动读取根目录 `rust-toolchain.toml`）

## 初始设置

```bash
bun setup
```

做了三件事：

1. `bun install` — 安装依赖
2. `bun run build:native` — 编译 Rust 原生模块
3. `bun --cwd=packages/coding-agent link && sh scripts/link-omp.sh` — 将 `omp` 链接到全局 PATH

### 已安装线上版 omp 的冲突处理

`bun setup` 中的 `bun link` 会**覆盖**全局 `omp` 链接，把生产版替换为本地 dev wrapper。

- **推荐：跳过 `bun setup`**，只用 `bun dev` 调试，不影响已安装的生产版
  ```bash
  bun install && bun run build:native
  bun dev   # 启动开发版，不改变全局 omp
  ```
- **需要 `omp` 命令指向本地源码**：跑 `bun setup`，之后 `omp` = 开发版
  恢复生产版：
  ```bash
  bun install -g @oh-my-pi/pi-coding-agent
  ```

## 项目结构

```
packages/coding-agent/   ← 主开发目标（omp CLI）
packages/ai/             ← 多 provider LLM 客户端
packages/agent/          ← agent 运行时（工具调用、状态管理）
packages/tui/            ← 终端 UI 库
packages/catalog/        ← 模型目录
packages/natives/        ← Rust 原生模块的 TS 绑定
crates/pi-natives/       ← Rust 原生模块源码
```

## 日常开发

在**项目根目录**执行：

| 操作                 | 命令                    | 说明                       |
| -------------------- | ----------------------- | -------------------------- |
| 启动 omp（开发模式） | `bun dev`               | 直接运行 TS 源码，无需编译 |
| 类型检查 + Lint      | `bun check`             | 提交前门禁                 |
| 自动修复             | `bun fix`               | biome + 格式化 prompts     |
| 运行全部测试         | `bun test`              | CI 用                      |
| 冒烟测试             | `bun run ci:test:smoke` | 快速验证 worker 通信       |

在 `packages/coding-agent/` 下也可以直接执行：

```bash
bun run check        # biome + tsgo
bun run check:types  # 仅类型检查
bun run test         # heavy 测试套件
bun run fix          # biome 自动修复
```

### 测试分层

```bash
bun run ci:test:coding-agent:singleton   # 单例测试
bun run ci:test:coding-agent:ui          # TUI 测试（chunk=5）
bun run ci:test:coding-agent:runtime     # 运行时测试
bun run ci:test:coding-agent:native      # 原生模块测试
bun run ci:test:coding-agent:heavy       # 完整测试
bun run test:rs                           # Rust 测试
```

### Rust 开发

```bash
bun run build:native   # 重新编译 native addon
bun run check:rs       # cargo fmt --check + clippy
bun run test:rs        # cargo nextest
```

## 启动流程

```
cli.ts → commands/ → main.ts → createAgentSession()
                                    ├── InteractiveMode (TUI)
                                    ├── runPrintMode (oneshot)
                                    └── runRpcMode (JSONL server)
```

`cli.ts` 也是 worker host：worker 通过 `__omp_worker_*` 隐藏 argv 重新进入同一入口模块。

## 关键约定

- **永远不要用 `tsc`** — 类型检查走 `bun check`（内部用 `tsgo`）
- **不要用 `console.log`** — 用 `logger` from `@oh-my-pi/pi-utils`
- **prompt 放 `.md` 文件** — 不在代码里拼字符串
- **worker 不创建独立入口** — 统一从 `cli.ts` 进入
- **barrel export 用 `export *`** — 不用命名重导出
- **类成员用 ES `#private`** — 不用 `private` 关键字

## 调试技巧

### 查看日志

```bash
tail -f ~/.omp/logs/omp.$(date +%Y-%m-%d).log
```

### 性能分析

```bash
PI_TIMING=x bun dev   # 模块加载耗时
```

### 冒烟测试（验证 worker）

```bash
bun packages/coding-agent/src/cli.ts --smoke-test
```

这会启动 stats sync worker 和 tiny-model 子进程，ping 后退出。

### MCP 调试

MCP 服务器从多个来源发现（配置优先级从高到低）：

- `~/.omp/agent/mcp.json`（用户级）
- `.omp/mcp.json`（项目级）
- `mcp.json` / `.mcp.json`（项目根）
- `~/.cursor/mcp.json`
- `~/.codex/config.toml`
- `~/.claude.json`
- 其他 IDE 配置

禁用一个服务器：

```json
// ~/.omp/agent/mcp.json
{ "disabledServers": ["computer-use"] }
```

查看 MCP 连接状态：启动 omp 后观察启动日志，或在 TUI 中用 `/mcp` 命令。
